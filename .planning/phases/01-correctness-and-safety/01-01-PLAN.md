**Part A: Fix import route (BUGS-02)**

Restructure `src/app/api/import/route.ts` to fix the critical issue: SQLite's `PRAGMA foreign_keys` is a no-op inside a transaction. The current code calls `db.pragma('foreign_keys = OFF')` INSIDE `db.transaction()`, so foreign keys are never actually disabled.

Changes to make:

1. **Move PRAGMA outside transaction boundary:**
   ```
   db.pragma('foreign_keys = OFF');  // BEFORE transaction
   try {
     const importTransaction = db.transaction(() => { ... });
     importTransaction();
   } finally {
     db.pragma('foreign_keys = ON');  // ALWAYS re-enable
   }
   ```

2. **Validate upfront before touching DB (per user decision):**
   - The existing `validateExportData()` function already validates structure. Keep it.
   - Add validation that each tote record has required fields (id, name, location) and each item has required fields (id, tote_id, name). Fail fast with clear error: "Import validation failed: item at index N is missing required field 'name'. No data was imported."

3. **Track written files for rollback cleanup (per user decision):**
   - Create a `writtenFiles: string[]` array before the try block
   - During ZIP photo extraction, push each written file path to `writtenFiles`
   - In the catch block, iterate `writtenFiles` and try to delete each (ignore individual unlink errors)

4. **Add path traversal protection for ZIP entries (VALID-03 connection):**
   - When extracting ZIP entries, resolve the full output path with `path.resolve()`
   - Verify the resolved path starts with the intended directory (`uploadsDir` or `thumbnailsDir`)
   - Skip entries that fail the check (log a warning)

5. **Keep existing photo file clearing** (lines 266-277) but move it AFTER successful transaction, before ZIP extraction. If the transaction fails, we don't want to have deleted existing photos.

6. **Restructure the overall flow:**
   ```
   Parse ZIP and JSON (existing code -- keep as-is)
   Validate export data structure (existing validateExportData)
   Validate individual records (new -- fail fast)
   db.pragma('foreign_keys = OFF')
   try {
     Run transaction (clear tables + insert all data)
   } finally {
     db.pragma('foreign_keys = ON')
   }
   Clear existing photo files from disk
   Extract new photo files from ZIP (with path traversal check, tracking writtenFiles)
   Return success
   ```
   Wrap the entire post-transaction file section in try-catch: if file extraction fails, clean up writtenFiles and return error.

**Part B: Fix photo DELETE route (BUGS-03)**

In `src/app/api/photos/[id]/route.ts`, the DELETE handler (lines 55-59) calls `fs.unlinkSync` without try-catch. If the file doesn't exist or permissions fail, the entire delete fails and the DB record is not removed.

Fix: Wrap each `fs.unlinkSync` call in its own try-catch. Log errors with `console.error`. Delete the DB record regardless of file cleanup success (per user decision: "log and continue" strategy).

Current problematic code:
```typescript
if (fs.existsSync(originalFile)) fs.unlinkSync(originalFile);
if (fs.existsSync(thumbnailFile)) fs.unlinkSync(thumbnailFile);
db.prepare('DELETE FROM item_photos WHERE id = ?').run(Number(id));
```

Change to:
```typescript
// Delete DB record first (this is the authoritative action)
db.prepare('DELETE FROM item_photos WHERE id = ?').run(Number(id));

// Clean up files (log and continue on failure)
try {
  if (fs.existsSync(originalFile)) fs.unlinkSync(originalFile);
} catch (fileErr) {
  console.error('Error deleting original photo file:', fileErr);
}
try {
  if (fs.existsSync(thumbnailFile)) fs.unlinkSync(thumbnailFile);
} catch (fileErr) {
  console.error('Error deleting thumbnail file:', fileErr);
}
```
